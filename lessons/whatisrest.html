REST چیست؟

ویدئو رو مخفی کن

معماری REST شش محدودیت تعیین می کند. این محدودیت ها را اولین بار "روی فیلدینگ" در پایان نامه دکتری مطرح کرد (لینک مقاله). این محدودیت های اصول توسعه با روش RESTful می باشد.

این 6 محدودیت عبارتند از (برای دیدن توضیحات بر روی گزینه ها کلیک کنید):

  • رابط یکسان و هماهنگ

    منظور رابط یکسان بین کلاینت و سرور است. این محدودیت معماری را ساده تر می کند و باعت می شود که کلاینت و سرور کاملا از هم جدا شوند و به صورت جداگانه توسعه داده شوند. برای برقراری این محدودیت باید چهار اصل زیر را رعایت کنیم:

    بر اساس منابع (resource-based)

     

    هر منبع با استفاده از URI قابل تشخص خواهد بود. به صورت مفهومی منبع با چیزی که به کلاینت نمایش داده می شود (representation) می تواند تفاوت داشته باشد. مثلا اگر کلاینت از سرور درخواست یک رکورد از دیتابیس را داشته باشد، سرور آن رکورد را مستقیم برای کلاینت ارسال نمی کند بلکه آن را در فرمت هایی مثل XML یا JSON به کلاینت ارسال می کند.

    تغییر منبع با استفاده از representation

     

    وقتی که اطلاعاتی با استفاده از representation به کلاینت داده می شود (شامل metadata)، کلاینت هر اطلاعاتی که برای تغییر یا حذف آن رکورد نیاز دارد را در اختیار دارد.

    پیام های توصیفی و شفاف (self-descriptive)

     

    یک پیام Self-descriptive یعنی در خود آن پیام تمامی اطلاعات مهم نهفته است یعنی برای پردازش یک پیام هر اطلاعاتی که لازم باشد در همان پیام موجود است. مثلا اینکه چه پارسری نیاز باشد که استفاده کنیم در خود  MIME type (https://stackoverflow.com/questions/3828352/what-is-a-mime-type)  مشخص می گردد.

    هایپرمدیا به عنوان موتور وضعیتی Hypermedia as the Engine of Application State (HATEOAS)

     

    کلاینت وضعیت را با استفاده از محتوای بدنه، پارامترهای query-string، هدر request و URI انتقال می دهد. سرویس وضعیت را با استفاده از محتوای بدنه، کد پاسخ (response code)، و هدر پاسخ انتقال می دهد. به این مفهوم هایپرمدیا می گویند (هایپرلینک در هایپرتکست).

    علاوه بر تعریف بالا HATEOS یعنی در صورت نیاز برای تامین اطلاعات به URI برای استخراج یک منبع، لینک ها در داخل بدنه یا هدر قرار بدهیم.

    رابط یکسان و هماهنگ یکی از اصول REST می باشد که باید در طراحی رعایت گردد.

  • فاقد حالت یا وضعیت (stateless)

    REST مخفف عبارت Representational State Transfer می باشد. Stateless بودن جز اصلی ترین خصوصیات REST است بگونه ای که اگر لازم باشد وضعیت کلاینت نگهداری شود اطلاعات آن در خود request خواهد بود مثلا در URI، query-string، بدنه یا هدر. URI مشخصه منبع (آدرس رکوردی که درخواست می شود) را نگهداری می کند و وضعیت (تغییر وضعیت) منبع در بدنه (Body) نگهداری می شود. سپس بعد از پردازش این درخواست توسط سرور، state در پاسخ توسط هدر، status یا بدنه پاسخ به کلاینت داده می شود.

    برای کسانی که سال ها در صنعت نرم افزار فعالیت کرده اند نگهداری state توسط session در تعداد زیادی از درخواست های HTTP امری جدید نیست. در REST اگر قرار باشد که state ای نگهداری شود، همه اطلاعات سمت کلاینت نگهداری می شود و در هر درخواست HTTP این اطلاعات را باید برای سرور ارسال کند. Stateless بودن امکان توسعه پذیری نرم افزار را به شدت افزایش می دهد چرا که دیگر سرور نیازی ندارد که اطلاعات session را نگه دارد و مدیریت کند. علاوه بر آن دیگر load balancer ها در سیستم های stateless نگران وابستگی های session ها نخواهند بود.

    تفاوت resource و state چیست؟ state چیزی است که برای سرور در مدیریت یک request مهم است مثل اطلاعات لازم در یک session. Resource همان داده ای است که در دیتابیس ذخیره شده است و بعد از یک request به صورت یک representation به کلاینت نشان داده می شود. State اطلاعاتی است که در هر request و برای هر کاربر می تواند متفاوت باشد. Resource اما داده ای است که در دیتابیس وجود دارد و هر کلاینت می تواند آن را درخواست کند.

  • قابل کش شدن (cacheable)

    یکی از قابلیت های وب قابلیت cache شدن است، کلاینت ها داده های پرکاربرد را cache می کنند بنابراین در response ها باید تعریف شود که اطلاعات این response بهتر است cache شود یا نه (بر اساس اهمیت و تکرار یک داده). همچنین مدیریت درست قابلیت cache شدن به صورتی که داده های غیرمهم را cache نکنیم یا اصلا قابلیت cache شدن را برداریم خودش به توسعه پذیری و کارایی سیستم کمک می کند.

  • کلاینت-سرور

    رابط یکسان (uniform interface) باعث استقلال کلاینت و سرور می شود. Separation of concerns مفهومی است به این معنی که با جداسازی هرچه بیشتر کلاینت و سرور بسیاری از مشکلات حل می شوند مثلا اینکه کلاینت دیگر دغدغه گرفتن داده از دیتابیس را ندارد و این قابلیت را به کلاینت می دهد که مستقل رشد کند و کار با دیتابیس محدود به سرور شود در عوض سرور اهمیتی به رابط کاربری یا وضعیت کاربر نیست بنابراین سرور ساده تر و توسعه پذیرتر می شود. حتی کلاینت یا سرور را می توان به راحتی با کلاینت یا سروری دیگر جاگزین کرد بدون اینکه در کارکرد دیگری اختلالی ایجاد شود.

  • سیستم لایه ای

    به صورت کلی کلاینت نمی داند که به سرور نهای متصل شده یا به یک سرور میانی. سیستم های میانی می تواند load balancer باشند یا سروری مخصوص cache اشتراکی باشند. این معماری شاید جهت افزایش امنیت تعریف گردند.

  • Code on Demand (optional)

    سرور می تواند گاهی به صورت موقتی بخشی از منطق برنامه را به کلاینت انتقال دهد مثل اسکریپت های جاوااسکریپت که در کلاینت اجرا می شوند.

    اگر سیستم خود را با این محدودیت های تعریف شده در معماری REST پیاده سازی کنیم موجب می شود که این سیستم های توزیع شده دارای قابلیت های زیادی باشد مانند: قابلیت توسعه پذیری، بهبود کارایی، سادگی، شفافیت، قابلیت پرتابل بودن و قابلیت اطمینان.

تنها محدودیت اختیاری در REST آیتم آخر یعنی code on demand است و اگر هر کدام دیگر از محدودیت ها رعایت نشود دیگه نمی توانیم به سیستم RESTful گفت.
